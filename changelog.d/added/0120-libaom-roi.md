- **libaom-av1 ROI steering** (`ffmpeg-patches/files/libaom-pelorus-roi.patch`, `-pelorus_roi 1`): patches `libavcodec/libaomenc.c` so `libaom-av1` honours `AV_FRAME_DATA_REGIONS_OF_INTEREST` — the standard ROI side data `vf_pelorus_analyze roi=1` emits — via libaom's segment-based `AOME_SET_ROI_MAP` (ADR-0114 Tier 1, the software-AV1 leg next to NVENC/QSV). Unlike the dense per-block delta-QP maps, libaom uses a segment model: ≤ 8 distinct qindex deltas per frame (`AOM_MAX_SEGMENTS`) over a 4×4-luma-cell grid. The bridge quantises each frame's region `qoffset`s into ≤ 8 segments (segment 0 = background), paints the cell grid by nearest segment (first region wins on overlap, like `libx264`), and pushes the map per frame before the encode. A single `pelorus_roi` AVOption (default off) exposes it; a frame with no ROI side data pushes nothing (zero behaviour change). **Verified on hardware** (built into ffmpeg n8.1.1 `--enable-libaom`, ran the analyze→libaom pipeline on a banding clip): no crash, valid output, the bridge builds a sane 120×68 grid and quantises 32 regions into 6 segments. **Honest negative on quality**: stock libaom 3.14.1's `AOME_SET_ROI_MAP` returns `AOM_CODEC_INVALID_PARAM` in FFmpeg's encoder configuration (the control is wired only for the RTC delta-q path upstream), so the map is ignored and CAMBI is identical with the option on or off (0.93193 both). The bridge degrades gracefully (one warning, no bias) and becomes effective for free once a libaom release enables the control on the path FFmpeg uses — an upstream limitation, not a patch defect. `generate.sh` extended to carry this libavcodec edit as patch 0012. See `docs/backends/libaom-roi.md` and ADR-0120.
