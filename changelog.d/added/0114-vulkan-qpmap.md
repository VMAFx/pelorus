- **Cross-vendor "via Vulkan" QP-map steering** (`ffmpeg-patches/files/vulkan-pelorus-qpmap.patch`, `-pelorus_roi 1`): patches `libavcodec/vulkan_encode.c` to drive `VK_KHR_video_encode_quantization_map`, so the proven `vf_pelorus_analyze roi=1` banding map (the standard `AV_FRAME_DATA_REGIONS_OF_INTEREST` side data) steers bit allocation on **every Vulkan-encode-capable GPU** — `h264_vulkan`/`hevc_vulkan`/`av1_vulkan` — with no host roundtrip (ADR-0114 Tier 2, the cross-vendor endgame). A single `-pelorus_roi` AVOption in the shared `VULKAN_ENCODE_COMMON_OPTIONS` macro exposes it on all three codecs. The map kind is driver-probed from the negotiated rate control: a signed **delta-QP map** (`R8_SINT`) under CQP, or an **emphasis map** (`R8_UNORM`) under CBR/VBR. Everything runtime-probes the extension + capability flag + texel size and degrades to a one-shot warning + pass-through; a frame with no ROI binds no map (zero behaviour change); the path is compile-gated by `VK_KHR_video_encode_quantization_map`. `generate.sh` extended to carry this libavcodec edit as patch 0008. **Compile-verified only** (`vulkan_encode.c` builds clean under FFmpeg's `-std=c17 -Wall` set; full stack replays onto pristine n8.1.1); on-HW proof is blocked on the dev box by a driver missing the encode-feedback flags (`BUFFER_OFFSET`/`BYTES_WRITTEN`) — a driver limitation, not a patch defect. No BD-rate gain claimed. ADR-0114.
